# HANDOFF: проект «Перехват» — киберучения по криптографии

Этот файл — конспект всей предыдущей работы над проектом, чтобы продолжить её в Claude Code
без пересказа истории заново. Положи его в корень папки `kiberperehvat/` (рядом с `README.md`)
и в Claude Code скажи что-то вроде: «прочитай HANDOFF.md, это контекст по проекту, продолжаем работу».

## Что это за проект

Игра-занятие по кибербезопасности для старшеклассников (15–17 лет), 7 команд по 4 человека,
1.5–2 часа, реальные ПК в одной локальной сети. Полное описание сценария (легенда, тайминг фаз,
роли внутри команды, система очков, темы дебрифа) — в отдельном файле `scenario_kiberbezopasnost.md`
(шёл отдельным файлом вне архива с кодом, спроси у пользователя, если понадобится — он его сохранил).

Коротко: 7 станций выстроены в линию и передают сообщение друг другу. По пути его один раз
шифруют, потом расшифровывают на другом конце; посередине сидит «крот», который слушает канал
и пытается взломать шифр; две станции по бокам от крота логируют трафик и считают хэши
(на случай, если крот не просто подслушивает, а подменяет сообщения).

## Текущее состояние: РАБОЧИЙ ПРОТОТИП, полностью протестирован

Код написан, гонялся через `test_harness.py` (полный e2e прогон всех 7 процессов по сети),

## Архитектура

**Топология** — симметричная линия из 7 узлов:
```
pk1(endpoint) - pk2(cipher) - pk3(database) - pk4(impostor) - pk5(database) - pk6(cipher) - pk7(endpoint)
```

**Важнейшее архитектурное решение:** зеркальные пары (pk1/pk7, pk2/pk6, pk3/pk5) выполняются
**одним и тем же кодом роли** — различие только в конфиге. Отсюда всего 4 роли, не 6-7:

- **`endpoint`** (pk1, pk7) — умеет и отправлять (`send`), и получать. У станции один сосед (`peer_host/peer_port`)
  и поле `send_direction` (`"right"` для pk1, `"left"` для pk7), которым помечаются исходящие сообщения.
- **`cipher`** (pk2, pk6) — шифрует или расшифровывает в зависимости от того, откуда пришло сообщение.
  Два соседа (`peer_left_*`, `peer_right_*`), плюс `cipher_type`/`cipher_key`/`encrypt_direction`.
  Шифрует, если направление сообщения совпадает с `encrypt_direction`, иначе расшифровывает тем же ключом.
  **Смена шифра — только через остановку процесса, правку конфига и перезапуск** (осознанное решение
  пользователя — это часть педагогики: «настроил → перезапустил → проверил» повторяется на каждой смене шифра,
  а не только в Фазе 0). Runtime-команды `set` больше нет.
- **`database`** (pk3, pk5) — логирует каждое проходящее сообщение (msg_id, hash, направление, содержимое),
  пробрасывает дальше без изменений. Два соседа.
- **`impostor`** (pk4) — копирует себе всё проходящее, обязан пробрасывать дальше (см. ниже про `tamper_next`).
  Два соседа.

**Каждое сообщение — JSON с полем `direction`** (`"right"` = от pk1 к pk7, `"left"` = от pk7 к pk1).
Любой узел получает сообщение с одной стороны и пересылает его в ТУ ЖЕ сторону, противоположному
соседу — вот и вся логика маршрутизации, она одинакова для database/impostor/cipher, различается
только тем, какие поля конфига считаются «соседями».

**Протокол связи** — JSON, один объект на строку (newline-delimited), поверх обычных TCP-сокетов.
Между станциями — короткие соединения (открыл → отправил → закрыл) на каждое сообщение.
Со станции до мастера — одно постоянное соединение (для статуса и live-ленты трафика).

## Структура файлов

```
common/protocol.py   — JSON-over-TCP хелперы, сборка сообщений (make_chain_message, make_master_report, ...)
common/crypto.py     — шифр Цезаря (кириллица+латиница+цифры), XOR (через base64), sha256-хэш для БД
station.py           — универсальный скрипт станции, вся ролевая логика в ROLE_HANDLERS/ROLE_COMMANDS
master.py            — консольная rich-панель мастера: статусы станций + live-лента трафика + event-инъекции
cipher_tool.py        — отдельный ручной шифратор/дешифратор для крота, использует common/crypto.py
                         напрямую, к станции/сети не подключается вообще (чистый калькулятор)
ask_master.py          — отдельный скрипт "спросить мастера", держит своё постоянное соединение
                         (регистрируется как register_ask, не как станция), весь диалог — в его окне
configs/*.json        — конфиги для локального теста на 127.0.0.1 (готовые примеры под каждую фазу шифра)
test_harness.py       — автотест: поднимает все 7 процессов + мастер, гоняет сообщения в обе стороны,
                         проверяет оба шифра, перехват, взлом Цезаря, tamper_next
README.md             — инструкция по разворачиванию на школьных ПК + справочник команд по ролям
```

## Что уже исправлено (не наступай на те же грабли)

**Баг мигания статуса станций (flapping) на панели мастера — ИСПРАВЛЕН.**
Причина была в двух местах одновременно:
1. `socket.create_connection(..., timeout=5)` — таймаут действовал не только на попытку подключения,
   но и на все последующие `recv()`. При простое (нет событий от мастера) станция ловила
   `socket.timeout` каждые 5 секунд и считала связь оборванной, переподключаясь заново, а старый
   сокет никогда явно не закрывался.
2. На мастере из-за этого возникали гонки: протухший поток от старой (уже неактуальной) регистрации
   мог с опозданием затереть статус свежей регистрации той же станции обратно на `offline`.

Фикс: `sock.settimeout(None)` сразу после установления соединения; явное закрытие сокета при любом
обрыве в `_reader`/`_writer`; heartbeat раз в 4 секунды со стороны станции; на мастере — сверка
`stations[sid]["socket"] is conn` перед тем как помечать станцию offline, и закрытие старого сокета
при новой регистрации того же station_id. Проверено 20-секундным простоем без единого лишнего реконнекта.

**Баг: `test_harness.py` падал на новой машине (`/home/evlakhov/study/kiberperehvat`) — ИСПРАВЛЕН.**
Два независимых момента:
1. В `test_harness.py` был захардкожен `ROOT = "/home/claude/kiberperehvat"` (путь из прошлого
   окружения). Заменено на `ROOT = os.path.dirname(os.path.abspath(__file__))` — тест теперь
   не зависит от абсолютного пути, где лежит проект.
2. Само сообщение `pk1 -> pk7` не находилось в выводе pk7 при прогоне теста. Причина: в `station.py`
   есть интерактивный `input()` в главном потоке консоли станции, а приём сообщения по цепочке
   печатается (`ПОЛУЧЕНО СООБЩЕНИЕ`, лог действия) из **фонового потока** (`_handle_conn`).
   Когда stdout станции не TTY (пайп — как в тестовом харнессе, или любой обёртке/редакторе),
   Python буферизует stdout блоками, и печать из фонового потока флашится только когда главный
   поток сам вызовет `input()` заново (флаш как побочный эффект печати промпта). Если станции
   никто не шлёт команд в её собственную консоль (как pk7, которая в тесте только принимает),
   вывод фонового потока может вообще не долететь до пайпа. В прошлом окружении (`/home/claude/...`)
   это, судя по всему, маскировалось глобальным `PYTHONUNBUFFERED=1`; на голой машине пользователя
   баг проявился. Фикс — `sys.stdout.reconfigure(line_buffering=True)` сразу после импортов в
   `station.py`, не полагаясь на переменные окружения. На реальных ПК (настоящий TTY) поведение
   не меняется — там Python и так включает line-buffering сам.

   Также выяснилось: если в рабочей директории остаются зависшие процессы `station.py`/`master.py`
   от прошлого прерванного прогона (например, тест упал раньше `finally`-блока или был убит не
   до конца), они держат TCP-порты (6001-6007, 7000) и следующий прогон теста будет падать с
   `OSError: Address already in use`. Перед прогоном на новой/незнакомой машине стоит проверить
   `ps aux | grep -E "station.py|master.py"` и убить лишнее.

**Мигание панели мастера каждые 1-2 секунды без реальных событий — ИСПРАВЛЕНО.**
Причина: каждая станция шлёт `heartbeat` раз в 4с (`station.py`, `_heartbeat_loop`), и на мастере
любое сообщение от станции — включая heartbeat — безусловно ставило `dirty.set()`, вызывая редроу
панели (после дебаунса 0.3с). При 7 станциях heartbeat приходит от кого-то из них в среднем
раз в ~0.5-1с, отсюда постоянное мигание, хотя в самой таблице ничего не менялось (heartbeat не
трогает ничего, кроме `last_ts`, который в панели не отображается).
Фикс в `master.py` (`handle_station`): `dirty.set()` теперь вызывается только когда что-то реально
изменилось — `register` и `log` всегда (новая станция/переподключение, новое действие в ленте),
`heartbeat` — только если статус станции до этого не был `online` (т.е. настоящий переход
offline -> online, а не просто подтверждение того, что станция и так на связи).

## Изменения по логам/выводу (запрошены пользователем, сделаны и протестированы)

1. **Убрана фаза/направление из консольного вывода станций** — раньше в каждой строке лога
   и в "ПОЛУЧЕНО СООБЩЕНИЕ"/"отправляю" мелькали `(фаза=..., направление=...)` — лишний шум для
   оператора станции на игре. `phase`/`direction` по-прежнему живут в самом сообщении и в
   `master_link.log()` не передаются вовсе (см. п.4) — они нужны только для маршрутизации,
   не для отображения.
2. **Шифраторы (pk2/pk6) теперь показывают контент до и после операции.** В `process_incoming`
   (station.py) добавлена проверка `outgoing != msg["content"]`: если хэндлер роли реально
   изменил содержимое (шифрование/расшифровка на cipher, а заодно и `tamper_next` на impostor —
   бесплатный бонус того же кода), в консоль печатается `до: «...» после: «...»` вместо просто
   `«...» -> {action}`. Для ролей без трансформации (database, cipher в режиме passthorugh)
   поведение не изменилось.
3. **pk1/pk7 (endpoint) — только выровняли под общий стиль** (убрали фазу/направление из
   "отправляю"/"ПОЛУЧЕНО СООБЩЕНИЕ"). Настоящего "до/после шифрования" там нет и не может быть —
   шифрование физически происходит только на pk2/pk6, эндпоинты его не видят. Уточнили этот момент
   с пользователем явно, прежде чем реализовывать.
4. **Панель мастера — облегчённая лента трафика, лимит 16 строк.** `master_link.log()` в
   station.py теперь шлёт `in_preview`/`out_preview` (что пришло / что ушло с этой станции)
   вместо старого набора `msg_id/phase/hop_from/direction/content_preview`. `action` тоже шлётся,
   но используется только в верхней таблице "Станции" (колонка "Последнее действие") — сама
   таблица статусов не трогалась, вопрос был именно про ленту трафика. Лента теперь:
   `{id_станции}: «вошло» -> «вышло»` (без ts/hop_from/фазы/id), `«—»` когда одной из сторон нет
   (например, у отправителя нет "вошло", у эндпоинта на приёме нет "вышло"). Последние 16 строк
   вместо 15 (`events = list(events)[-16:]`, заголовок панели тоже поправлен).

## Новые фичи по запросу пользователя (сделаны и протестированы)

**Запрос подсказки мастеру — сначала команда `ask` в station.py, ПОТОМ вынесена в отдельный
скрипт `ask_master.py` по прямому запросу пользователя.** На мастере (`master.py`) приём не
менялся вообще: отдельная ветка `mtype == "hint_request"` в `handle_station`
(`hint_requests = deque(maxlen=50)`), не смешана с обычной лентой трафика (`events`) — выводится
отдельной жёлтой панелью «Запросы подсказок (последние 10)». Отвечает мастер обычным
`event <id_станции> <текст>`, никакого механизма подтверждения/закрытия запроса нет (осознанно —
не просили).

Первая версия была командой `ask <текст>` прямо в консоли станции (глобальная, не через
`ROLE_COMMANDS`, потому что нужна на любой роли) — но пользователь попросил вынести её в отдельный
скрипт, чтобы набор вопроса не путался с живой лентой входящих сообщений станции (та печатается
из фонового потока в произвольный момент, в том числе посреди набора текста в консоли). Сделано:
- `MasterLink.ask()` в station.py УДАЛЁН вместе с командой `ask` в `console_loop` — станция теперь
  умеет только `state`/`help`/`quit` + свои ролевые команды, про подсказки мастеру ничего не знает.
  Баннер станции при старте выводит подсказку с точным путём до своего конфига:
  `python3 ask_master.py --config {путь_к_конфигу_этой_станции}` — для этого при загрузке конфига
  в `main()` добавили `config["_config_path"] = args.config`, чтобы баннер знал реальный путь.
- Новый файл `ask_master.py` — полностью автономный, к station.py и MasterLink не имеет отношения.
  Каждый вопрос уходит ОТДЕЛЬНЫМ коротким соединением (`protocol.send_chain_message` — то же самое,
  чем станции шлют сообщения по цепочке: открыл -> отправил -> закрыл), без регистрации на мастере
  и без постоянного соединения. Это важно и осознанно: `master.py` при новой `register` с уже
  занятым `station_id` ЗАКРЫВАЕТ старое соединение (см. фикс мигания статуса выше в этом файле) —
  если бы `ask_master.py` регистрировался тем же `station_id`, что и сама станция, он бы обрывал
  постоянное соединение станции с мастером при каждом запуске. Проверено, что это не так: `hint_request`
  на мастере обрабатывается БЕЗ проверки, что `station_id` вообще зарегистрирован (просто
  `hint_requests.append(msg)`), так что отдельный процесс может слать вопросы вообще не регистрируясь.
- Проверено вручную (debug-скрипт, не часть `test_harness.py` — тут нечего особо регресс-тестировать,
  никакой сложной логики): подняли master+pk3, отдельным процессом `ask_master.py --config
  configs/pk3.json` отправили два вопроса — оба появились на панели мастера с правильным `station_id`,
  а pk3 всё это время оставался `online` (соединение станции не разорвалось).

**Версия 2: весь диалог (вопрос+ответ) внутри окна ask_master.py, а не разбросан по двум окнам.**
Версия 1 выше слала вопрос коротким соединением и на этом останавливалась — ответ мастера
(`event <id_станции> <текст>`) уходил на единственное известное мастеру соединение для этого
`station_id`, то есть в консоль САМОЙ СТАНЦИИ, а не в окно, откуда задали вопрос. Пользователь
явно попросил это поправить: диалог должен быть целиком в одном окне. Ключевая сложность та же,
что и раньше — `ask_master.py` НЕЛЬЗЯ регистрировать тем же `station_id`, что и станцию (см. v1
выше про вытеснение соединения). Решение — завести на мастере ВТОРОЙ, полностью отдельный реестр:
- `master.py`: новый словарь `ask_sockets` (`station_id -> [socket, ...]`), независимый от
  `stations`. Новый тип отчёта `register_ask` (просто `msg["station"]`, без роли/display_name —
  это не станция, в таблицу "Станции" не попадает и не обязано быть онлайн/оффлайн). В
  `handle_station` — если пришёл `register_ask`, добавляем сокет в `ask_sockets[station_id]`;
  при закрытии соединения — убираем оттуда (симметрично тому, как `stations[id]` гасится в
  `offline`, только тут просто удаление из списка, статуса у ask-окон нет).
- `send_event(target, text)` — переработана: для `target == "all"` поведение НЕ ТРОНУТО
  (широковещательные объявления идут только реальным станциям, ask-окна их не видят — это
  разные каналы по смыслу). Для конкретного `target`: если для этого `station_id` сейчас
  открыто хотя бы одно ask-окно — событие уходит ТУДА и ТОЛЬКО туда (`items = []`, станция
  ничего не получает); если ask-окон нет — как раньше, событие идёт станции. Один и тот же
  оператор-мастер команда `event <id> <текст>`, поведение выбирается автоматически по тому,
  открыто ли для этого id окно вопросов — мастеру думать об этом не нужно.
- `ask_master.py` переписан с одноразовых соединений на постоянное: класс `AskLink`
  (структурно копия `MasterLink` из station.py — `_connect_loop`/`_writer`/`_reader` через
  очередь и фоновые потоки, реконнект при обрыве), но БЕЗ heartbeat (ask-окна не участвуют в
  таблице статусов, незачем). Регистрируется `register_ask`, вопросы шлёт через очередь на этом
  же соединении, входящие `event` печатает прямо в своё окно. Не забыли и здесь: без
  `sys.stdout.reconfigure(line_buffering=True, encoding="utf-8")` ответ мастера, напечатанный из
  фонового потока `_reader`, завис бы в буфере до следующего `input()` — та же ловушка, что чинили
  в station.py, актуальна для любого нового скрипта с фоновым потоком + `input()` в главном.
- Проверено вручную (debug-скрипт): подняли master + реальную станцию pk4 + отдельно
  `ask_master.py --config configs/pk4.json`. Вопрос из окна ask появился на панели мастера.
  `event pk4 <текст>` пришёл ИМЕННО в окно ask (не в консоль pk4). `event all <текст>` пришёл в
  консоль pk4 как обычно и НЕ попал в окно ask — оба канала ведут себя независимо, как задумано.

**Задержка 2 секунды перед каждой пересылкой на следующий узел.** Реализовано в одной точке —
`forward()` в `station.py` (`time.sleep(FORWARD_DELAY_SECONDS)` перед `send_chain_message`),
поэтому действует одинаково при `send` с эндпоинта и при пересылке через cipher/database/impostor.
Значение берётся из `os.environ.get("PEREHVAT_FORWARD_DELAY", "2")` — **важно**: `test_harness.py`
явно переопределяет её в `0` при запуске станций (`STATION_ENV`), иначе один полный прогон
pk1->pk7 (6 хопов) занимал бы 12+ секунд вместо мгновенного, и вся автоматическая проверка
растягивалась бы на полторы-две минуты. Если понадобится погонять тест именно с реальной
задержкой (например, чтобы руками проверить, что она ощущается на занятии) — запускай station.py
напрямую без этой переменной, дефолт 2 секунды сработает сам.

**Прямая связь pk3<->pk5 (в обход цепочки) — было автоматической сверкой, ПЕРЕДЕЛАНО в свободный
обмен текстом по прямому запросу пользователя.** Первая версия была командой `sync`: инициатор
слал свой журнал (`msg_id`+`hash`), получатель сам сравнивал и подсвечивал расхождения — то есть
сверку делал код. Пользователь явно сказал, что это не нравится, и попросил вместо этого просто
дать pk3/pk5 возможность переписываться напрямую, а сверку хэшей операторы делают сами глазами.
Текущая реализация (`db_message_send`/`handle_db_message`, команда `msg <текст>` у роли `database`):
- Поля конфига `db_peer_host`/`db_peer_port` (`configs/pk3.json`/`configs/pk5.json`) остались теми
  же — адрес ЗЕРКАЛЬНОЙ БД, не соседа по цепочке.
- Протокол — тот же приём (`send_chain_message`, JSON-построчно), сообщение помечено
  `"type": "db_message"` (было `"db_sync"`) с единственной полезной нагрузкой `"text"` — просто
  свободная строка, никакой структуры под сравнение хэшей в протоколе больше нет. `_handle_conn` в
  `station.py` смотрит на `msg.get("type") == "db_message"` и просто печатает
  `>>> СООБЩЕНИЕ ОТ {from}: {text}` — вообще без сравнения с собственным журналом.
- Никакой блокировки/`state_lock` тут больше не нужно вообще — `handle_db_message` ничего не читает
  и не пишет в `state`, только печатает. Это сильно проще прежней версии (которая гоняла `state_lock`
  и в отправке, и в приёме, и требовала аккуратности с реентерабельностью лока).
- Идея сверки не изменилась и по-прежнему верна (важно понимать, если будешь что-то менять):
  каждое сообщение проходит через ОБЕ БД по разные стороны от крота pk4 с одним и тем же `msg_id`,
  так что расхождение хэша одного `msg_id` между `log` на pk3 и pk5 — прямой признак подмены кротом.
  Просто теперь операторы сверяют это глазами через `msg`, а не автоматика.
- `test_harness.py` обновлён под новую команду: `msg` от pk3 к pk5 и обратно, проверяем, что текст
  долетает в обе стороны (`>>> СООБЩЕНИЕ ОТ pk3/pk5: ...`). Старая проверка на `sync`
  (совпало/расходится) удалена вместе с самой командой.

**Совместимость с Windows — реальный баг, найден и исправлен.** Пользователь спросил, будет ли всё
это работать на Windows (школьные ПК). Код кросс-платформенный (чистый stdlib + rich), НО: символы
`‼`/`…`/«ёлочки», которые уже были в коде, не входят в старые кодировки консоли Windows (cp866/cp1251,
дефолт в обычном `cmd.exe`) — `print()` такого символа кидает `UnicodeEncodeError` и роняет процесс.
На Linux (где всё до сих пор тестировалось) это никогда не всплывало, потому что там UTF-8 по умолчанию.
Фикс — явное `sys.stdout.reconfigure(encoding="utf-8")` в начале `station.py` и `master.py` (было уже
`reconfigure(line_buffering=True)` в station.py для другого бага — просто добавили `encoding=` тем же
вызовом). Это гарантирует, что кодирование не упадёт, но НЕ гарантирует, что сам терминал ОТОБРАЗИТ
кириллицу правильно — это дело терминала (см. README, раздел "Если ПК на Windows": рекомендация
использовать Windows Terminal/PowerShell, не старый cmd.exe). Заодно поправили `test_harness.py`:
`subprocess.Popen(..., encoding="utf-8")` явно, а не полагаться на `text=True` без указания кодировки —
на Windows-хосте это брало бы кодировку локали, которая может не совпасть с тем, что реально шлёт
дочерний процесс, и ловило бы `UnicodeDecodeError` на стороне самого харнесса.

**Ручной режим у крота (impostor) — новая механика, замена/дополнение к tamper_next.**
Мотивация пользователя: с `tamper_next` крот заранее вслепую "заряжает" подмену для следующего
сообщения, не видя его — приходится гадать с таймингом. В ручном режиме крот сам решает судьбу
каждого сообщения, уже видя его содержимое.

- `state["manual_mode"]` (bool, по умолчанию falsy/выкл) + `state["pending"]` (dict `msg_id -> msg`,
  полное оригинальное сообщение цепочки, а не урезанная запись).
- `h_impostor` (station.py): если `manual_mode` включён — сообщение уходит в `pending`, хэндлер
  возвращает `(None, {}, action, None)`, то есть `process_incoming` НЕ вызывает `forward()` — вот и
  вся задержка, никакого отдельного планировщика/таймера не потребовалось.
- Новые команды роли `impostor`: `manual [on|off]`, `pending`, `release <id> [текст]`,
  `release_all`. `release` без текста — форвардит `m["content"]` как есть; с текстом — использует
  его вместо оригинала (это теперь основной способ подмены в ручном режиме, `tamper_next` в этом
  режиме сознательно отключён — печатает подсказку использовать `release` вместо себя).
- **Важно, не сломать при доработках**: и `_impostor_release_one`, и остальные консольные хэндлеры
  вызываются из `console_loop` внутри `with state_lock:` — сама `_impostor_release_one` НЕ берёт
  лок повторно (иначе дедлок, `threading.Lock` в проекте не реентерабельный — если пишешь новый
  консольный хэндлер, который сам лезет в `state_lock`, помни про это). `forward()` внутри тоже не
  трогает `state_lock`, только сетевой I/O.
- Если `manual off` вызван при непустом `pending` — сообщения НЕ отпускаются автоматически, только
  предупреждение в консоль (сколько их и что нужно `release`/`release_all` руками). Осознанное
  решение: неявный автосброс при выключении режима был бы более "удобным", но менее предсказуемым
  и мог бы случайно доставить сообщение в момент, когда оператор ещё не решил, что с ним делать.
- Проверено и закреплено в `test_harness.py` (сразу после Фазы 0, пока pk2/pk6 ещё в режиме
  `open` — специально ДО первого `restart_cipher_pair`, чтобы шифр не мешал сверять текст один в
  один): `manual on` → сообщение не доходит само → `pending` находит его → `release <id> <текст>`
  доставляет именно текст подмены, не оригинал → `manual off` восстанавливает автоматику.

**`cipher_tool.py` — отдельный ручной шифратор/дешифратор для крота.** Логичное продолжение
ручного режима выше: крот в ручном режиме видит перехваченный текст и хочет подготовить
правдоподобную подмену (или расшифровать перехваченное) — этот скрипт просто прогоняет текст
через `common/crypto.py` (тот же код, что и станции-шифраторы pk2/pk6, поэтому результат
гарантированно совпадает 1:1 с тем, что дала бы настоящая станция). Полностью автономный —
не подключается ни к сети, ни к station.py, чистый CLI-калькулятор. Поток: выбрать алгоритм
(1=Цезарь/2=XOR) → ключ → e/d (зашифровать/расшифровать) → текст → результат → зациклено (`s`
меняет алгоритм/ключ, `q` выход, Ctrl+D/Ctrl+C — тоже аккуратный выход, без трейсбека). Проверено
вручную через `printf ... | python3 cipher_tool.py` — Цезарь(13) и XOR('A') на тех же строках,
что использовались в `test_harness.py`, дали БАЙТ-В-БАЙТ те же шифротексты, что реально шли по
сети в тестах — подтверждает, что это не отдельная переизобретённая реализация, а тот же crypto.py.

**Баг: длинные сообщения обрезались многоточием у крота — ИСПРАВЛЕН.** `preview(text, n)` в
`station.py` изначально применялась ко ВСЕМУ подряд, включая локальный вывод на консоль станции —
не только к тому, что реально уходит на компактную панель мастера. Из-за этого живой лог входящих
(`process_incoming`), `list` и `pending` у крота обрубали контент до 60/80 символов с `…`, хотя
именно кроту critически важно видеть сообщение целиком, чтобы решить, подменять его или нет.
Фикс: разделили два разных места, где раньше использовалась одна и та же `preview()`:
- Локальный `print()` на консоль станции (в `process_incoming`, `cmd_impostor_list`,
  `cmd_impostor_pending`) — теперь показывает содержимое БЕЗ обрезки, как есть.
- Отчёт на панель мастера (`master_link.log(..., in_preview=..., out_preview=...)`) —
  по-прежнему через `preview()`, компактность там осталась намеренной (см. выше запрос
  пользователя «менее подробный вывод»/лимит 16 строк на панели мастера).

  Так что если снова понадобится урезать/расширить вывод — не трогай `preview()` бездумно
  глобально, там теперь два разных потребителя с разными требованиями (полнота на консоли
  станции vs компактность на панели мастера), их стоит различать явно.
- `cmd_impostor_show`/`cmd_db_log` уже показывали полное содержимое (и не трогались) — `show`
  никогда не был обрезан, только `list`/`pending`/живой лог.
- Проверено вручную (debug-скрипт, не `test_harness.py`): сообщение в 221 символ дошло целиком
  и без `…` во всех трёх местах — живой лог pk4, `list`, `pending`.

## Особенности поведения, которые выглядят как баг, но им не являются

- Если крот (`tamper_next`) подменяет сообщение НЕ зашифрованным текстом, а на приёмнике активен шифр —
  дешифратор всё равно попытается его расшифровать и получатель увидит нечитаемую кашу, а не подставленный
  текст дословно. Это реалистичное и осознанно оставленное поведение (подмена внутри шифрованного канала
  без знания ключа выглядит именно так) — своего рода бонусный урок для команды получателя: каша на экране —
  сама по себе сигнал тревоги.

**Раздаточные инструкции для команд по запросу пользователя — сначала один файл `SETUP.md`,
ПОТОМ разбит на 7 отдельных файлов по запросу.** Отдельно от README: README — техдокументация
для разработчика/ведущего, эти файлы — то, что реально даётся командам на занятии.

Версия 1 была одним `SETUP.md`, организованным ПО ТИПАМ станций (endpoint/cipher/database/
impostor), но внутри каждого типа — оба физических узла (например, отдельно ПК2 и ПК6), поскольку
`send_direction`/`encrypt_direction`/порты/`db_peer` у зеркальной пары различаются. Пользователь
попросил разбить инструкции отдельно по файлам — теперь:
- `setup/pk1.md` … `setup/pk7.md` — по одному файлу НА КАЖДУЮ ФИЗИЧЕСКУЮ СТАНЦИЮ (не на тип),
  каждый полностью самодостаточен: не отсылает к другим файлам, не требует читать README —
  всё нужное (что поставить, шаблон конфига, шаги запуска, команды именно этой роли, `ask_master.py`,
  troubleshooting) продублировано внутри. Осознанно избыточно по контенту между файлами — это
  раздатка для печати командам на разных ПК, а не код, DRY тут не приоритет.
  Практическая причина решения делать по физической станции, а не по типу: так можно выдать
  команде РОВНО её файл и ничего лишнего (в топологии есть пары с разными правами/возможностями —
  например, крот ПК4 не должен получить руки на команды других ролей просто потому что читает
  общий документ).
- `SETUP.md` в корне стал коротким индексом-таблицей (станция → роль → ссылка на `setup/pkN.md`),
  не дублирует контент.
- В шаблонах каждого файла заполнено всё, кроме IP-адресов (`<IP станции ПКx>`, `<IP ведущего>`) —
  это то, что команды получают от ведущего на занятии и вписывают сами; `listen_host` везде
  `0.0.0.0` (не `127.0.0.1` — принципиально для реального развёртывания на разных ПК, в отличие
  от `configs/*.json`, которые для локального теста на одной машине).
- Проверено: все 7 JSON-шаблонов (`json.loads`) валидны, и каждый ПОЛЕ-В-ПОЛЕ сверен скриптом
  с соответствующим `configs/*.json` (кроме намеренных отличий: `listen_host` и IP-плейсхолдеров) —
  расхождений нет.
- Если будешь менять схему конфига (добавлять/переименовывать поля роли) — не забудь синхронно
  поправить ВСЕ 7 файлов в `setup/`, они не генерируются автоматически из кода, это ручные
  документы, и теперь их семь, а не один — легко забыть один из них при правке.

**Версия 3: из `setup/pk*.md` убрано ВСЁ, кроме описания роли и списка команд.** Пользователь
сначала попросил обсудить структуру (я предложил компактный quick-start сверху + справочник
снизу, под печать на одну страницу), но в итоге запрос оказался жёстче: убрать вообще всё, кроме

процедура смены шифра для cipher (осталась только одна поясняющая фраза в описании роли — что
рантайм-команды нет и меняется через конфиг+рестарт, без пошаговых инструкций), секции про
`cipher_tool.py` и `ask_master.py`, troubleshooting. Итог — сеть/конфиги/вспомогательные скрипты

  «Точки Б», «Запуститe» с латинской e) — НЕ трогал их, это не моя правка, специально оставлено
  как есть по инструкции харнесса не отменять чужие интенциональные правки без явной просьбы.

## Идеи для дальнейшей работы (озвучены, но не реализованы)

  сдвигов уже есть).

  может визуально «съедать» вводимую в этот момент команду (это описано как известное ограничение в README,
  не баг, просто UX-компромисс ради простоты инфраструктуры).

## Как проверить, что всё работает, после любых изменений

```bash
cd kiberperehvat
python3 test_harness.py
```
Должно закончиться строкой `=== ВСЕ ПРОВЕРКИ ПРОШЛИ УСПЕШНО (включая оба направления) ===`.
Если меняешь `station.py` или `master.py` — обязательно прогоняй этот тест перед тем как отдавать
пользователю новую версию, он ловит регрессии в маршрутизации, шифрах и связи с мастером.
