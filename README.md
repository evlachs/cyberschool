# «Перехват» — киберучения по криптографии

Код к сценарию (`scenario_kiberbezopasnost.md`). Требуется Python 3.9+.

Этот файл — техническая документация проекта. Раздаточные карточки для команд на занятии (легенда
роли, задача, шаги запуска, команды) — см. [`SETUP.md`](SETUP.md).

## Топология и роли

Симметричная линия из 7 узлов:
```
pk1(endpoint) - pk2(cipher) - pk3(database) - pk4(impostor) - pk5(database) - pk6(cipher) - pk7(endpoint)
```
Зеркальные пары (pk1/pk7, pk2/pk6, pk3/pk5) используют **один и тот же код роли** — различие
только в конфиге. Каждое сообщение несёт поле `direction` (`right` — от pk1 к pk7, `left` — от pk7 к pk1);
любой узел получает сообщение с одной стороны и пересылает его дальше в ту же сторону, противоположному соседу.

Роли: `endpoint` (pk1, pk7 — умеют и отправлять, и получать), `cipher` (pk2, pk6 — шифруют/расшифровывают
в зависимости от направления), `database` (pk3, pk5), `impostor` (pk4).

## Что где

```
common/protocol.py   — JSON-over-TCP протокол связи
common/crypto.py     — шифр Цезаря, XOR, хэш целостности
station.py           — универсальный скрипт станции (запускается на всех 7 ПК команд)
master.py            — панель мастера игры (запускается на ПК ведущего)
cipher_tool.py        — отдельный ручной шифратор/дешифратор для крота (ПК4), не часть станции
ask_master.py          — отдельный скрипт "спросить мастера", запускается в соседнем окне, не часть станции
configs/*.json        — примеры конфигов для локального теста (localhost)
test_harness.py       — автотест всей цепочки в обе стороны (для проверки перед занятием)
SETUP.md              — индекс раздаточных карточек (кто где, какой configs/<роль>.json)
setup/pk1.md ... pk7.md — по одной карточке на станцию: легенда, задача, шаги запуска, команды
```

Карточки в `setup/` ссылаются на `configs/endpoint.json` / `cipher.json` / `database.json` /
`impostor.json` (по роли, не по номеру станции) — эти файлы-шаблоны с пропущенными полями под
IP нужно подготовить и выдать командам отдельно, в репозитории их пока нет (см. `configs/*.json`
для справки, какие поля у какой роли).

## Подготовка (за день до занятия)

1. На ПК ведущего: `pip install rich`
2. На всех 7 ПК команд: ничего ставить не нужно, только Python 3.
3. Убедитесь, что ПК в одной локальной сети и файрвол не блокирует выбранные порты (по умолчанию 6001–6007 для станций, 7000 для мастера).
4. Заранее узнайте IP-адреса всех ПК (`ipconfig` в Windows / `ip a` в Linux/Mac) — они понадобятся для заполнения конфигов на Фазе 0.

### Если ПК на Windows

Код кросс-платформенный (чистый stdlib + `rich`), но есть нюансы:
- Команда называется `python`, не `python3` (если стоит запускник из Microsoft Store — может быть и `python3`, проверьте `python --version` / `python3 --version` заранее).
- Запускайте из **Windows Terminal** или PowerShell, а не старого `cmd.exe` — там лучше поддержка Unicode и цветного вывода `rich`. Кириллица и служебные символы (`‼`, `…`, «ёлочки») теперь явно выводятся в UTF-8 из кода (иначе на устаревшей консольной кодировке Windows это роняло бы станцию с ошибкой кодировки) — но само окно терминала должно уметь ЭТО ОТОБРАЖАТЬ, отсюда и совет про Windows Terminal.
- При первом запуске `station.py`/`master.py` Windows Defender Firewall почти наверняка спросит разрешение на входящие подключения — обязательно разрешить, иначе станция не будет видна соседям в цепочке.

## Разворачивание на занятии

### Ведущий
```
python3 master.py --port 7000
```
Дашборд покажет статус станций и live-ленту трафика по мере их подключения.

### Каждая команда (Фаза 0 сценария)
1. Скопировать `station.py`, папку `common/` и получить свой `config.json` (или заполнить шаблон).
2. Открыть/создать `config.json`. Схема зависит от роли:

**endpoint** (ПК1, ПК7) — один сосед:
```json
{
  "station_id": "pk1",
  "role": "endpoint",
  "display_name": "Точка А",
  "listen_host": "0.0.0.0",
  "listen_port": 6001,
  "peer_host": "<IP соседнего узла>",
  "peer_port": 6002,
  "send_direction": "right",
  "master_host": "<IP ведущего>",
  "master_port": 7000
}
```
   `send_direction`: `"right"` для ПК1 (сообщения идут к ПК7), `"left"` для ПК7 (сообщения идут к ПК1).

**cipher** (ПК2, ПК6) — два соседа + параметры шифра:
```json
{
  "station_id": "pk2",
  "role": "cipher",
  "display_name": "Шифроузел A",
  "listen_host": "0.0.0.0",
  "listen_port": 6002,
  "peer_left_host": "<IP узла слева>", "peer_left_port": 6001,
  "peer_right_host": "<IP узла справа>", "peer_right_port": 6003,
  "cipher_type": "open",
  "cipher_key": null,
  "encrypt_direction": "right",
  "master_host": "<IP ведущего>",
  "master_port": 7000
}
```
   `encrypt_direction` — направление, в котором этот узел ШИФРУЕТ (в противоположном — расшифровывает).
   Для ПК2 (слева — ПК1, "своя" сторона) это `"right"`. Для ПК6 (справа — ПК7, "своя" сторона) — `"left"`.
   Ровно один и тот же код `station.py` с ролью `cipher` обслуживает обе станции — отличается только конфиг.

**database** (ПК3, ПК5) — два соседа по цепочке + прямая связь с зеркальной БД, в обход цепочки:
```json
{
  "station_id": "pk3",
  "role": "database",
  "display_name": "База данных A",
  "listen_host": "0.0.0.0",
  "listen_port": 6003,
  "peer_left_host": "<IP узла слева>", "peer_left_port": 6002,
  "peer_right_host": "<IP узла справа>", "peer_right_port": 6004,
  "db_peer_host": "<IP другой БД>", "db_peer_port": 6005,
  "master_host": "<IP ведущего>",
  "master_port": 7000
}
```
   `db_peer_host`/`db_peer_port` — адрес ЗЕРКАЛЬНОЙ БД (для ПК3 это ПК5, и наоборот), не соседа
   по цепочке. Каждое сообщение проходит через обе БД по разные стороны от крота ПК4, поэтому
   операторы ПК3/ПК5 могут напрямую списаться друг с другом (см. `msg` ниже) и сверить хэш
   конкретного `msg_id` вручную — расхождение выдаёт подмену содержимого кротом.

**impostor** (ПК4) — два соседа, без параметров шифра и без прямой связи с БД:
```json
{
  "station_id": "pk4",
  "role": "impostor",
  "display_name": "Крот",
  "listen_host": "0.0.0.0",
  "listen_port": 6004,
  "peer_left_host": "<IP узла слева>", "peer_left_port": 6003,
  "peer_right_host": "<IP узла справа>", "peer_right_port": 6005,
  "master_host": "<IP ведущего>",
  "master_port": 7000
}
```

3. Запустить:
```
python3 station.py --config config.json
```
4. Дождаться сообщения о подключении к мастеру, проверить связь тестовым `ping` в обе стороны (см. ниже).

## Команды по ролям (вводятся в консоли станции)

Общие для всех: `state`, `help`, `quit`.

Спросить подсказку у мастера — не команда в консоли станции, а отдельный скрипт `ask_master.py`,
запускаемый в СОСЕДНЕМ окне терминала (см. раздел ниже) — чтобы вопрос мастеру не перемешивался
с живой лентой входящих сообщений станции.

Каждая пересылка на следующий узел (при `send`, а также при прохождении через
cipher/database/impostor) идёт с задержкой 2 секунды — чтобы участники видели, что
сообщение реально идёт по цепочке шаг за шагом, а не оказывается на другом конце мгновенно.
На полном маршруте (6 хопов от ПК1 до ПК7) это, соответственно, ~12 секунд в одну сторону —
это ожидаемо, не зависание.

**ПК1/ПК7 — endpoint** (обе станции — одна и та же роль, могут и отправлять, и получать)
```
send <текст>     — отправить сообщение соседу (направление задано в конфиге)
phase <имя>      — пометить текущую фазу (open / cipher1 / keyexchange / cipher2) — для наглядности в логах
inbox [n]        — показать последние n полученных сообщений
```

**ПК2/ПК6 — cipher** (обе станции — одна и та же роль)

Нет runtime-команды смены шифра. Тип и ключ шифра задаются полями
`cipher_type` (`open` / `caesar` / `xor`), `cipher_key` и `encrypt_direction` в `config.json`
и применяются только при запуске. Смена шифра по ходу игры — это:

1. Остановить станцию (`quit` в консоли или Ctrl+C)
2. Отредактировать `config.json`: поменять `cipher_type`/`cipher_key` на новые (согласованные с зеркальной станцией)
3. Запустить заново: `python3 station.py --config config.json`

Так смена шифра всегда идёт через тот же паттерн «настроил конфиг → перезапустил → проверил», что и в Фазе 0.
В `configs/` есть примеры `pk2_caesar_example.json`, `pk2_xor_example.json` и аналогичные для pk6 — их можно взять за основу.
Поскольку узел шифрует в одну сторону и расшифровывает в другую (по полю `direction` в сообщении),
одна и та же станция автоматически обслуживает трафик, идущий в обе стороны.

**ПК3/ПК5 — database**
```
log [n]      — показать последние n записей журнала (msg_id, hash, направление, содержимое)
msg <текст>  — отправить сообщение зеркальной БД (в обход цепочки, через db_peer_host/db_peer_port)
```
`msg` — просто прямой текстовый канал между ПК3 и ПК5, без какой-либо автоматической сверки:
операторы сами решают, чем обменяться и как сравнивать (например, один диктует хэш конкретного
`msg_id` из своего `log`, второй сверяет со своим глазами). Сообщение приходит мгновенно — этот
канал не подчиняется общей задержке 2с на пересылку по цепочке (это отдельная, служебная связь).
Расхождение хэша одного и того же `msg_id` между ПК3 и ПК5 — прямое доказательство, что крот ПК4
подменил содержимое в промежутке между ними, а не просто подслушал (см. также `log` у обеих БД).

**ПК4 — impostor**
```
list [n]                    — перехваченные сообщения (обе стороны)
show <msg_id>                — показать сообщение целиком
crack_caesar <id> [сдвиг]    — расшифровать конкретным сдвигом, либо перебрать все 1..32
crack_xor <id> <ключ>        — попробовать расшифровать XOR предполагаемым ключом
tamper_next <текст>          — подменить следующее входящее сообщение перед пересылкой
                                (только в автоматическом режиме, см. ниже)
manual [on|off]               — ручной режим: входящие НЕ пересылаются сами, ждут вашего release
pending                       — показать сообщения, которые ждут решения (ручной режим)
release <id> [текст]          — отпустить конкретное сообщение: без текста — как пришло,
                                 с текстом — подмена именно этим текстом
release_all                   — отпустить все ожидающие без изменений (аварийный клапан,
                                 чтобы случайно не заблокировать игру насовсем)
```
В автоматическом режиме (по умолчанию) крот пересылает всё сразу, как и раньше — `tamper_next`
заранее «заряжает» подмену для следующего сообщения вслепую. В ручном режиме (`manual on`)
входящие сообщения ОСТАНАВЛИВАЮТСЯ на ПК4 и ждут вашей команды `release` — можно сначала посмотреть
содержимое (`pending`, `show <id>`), а потом решить: отпустить как есть или подменить конкретным
текстом именно в тот момент, когда это выгоднее всего. Не забывайте про `release_all`, если
увлеклись и накопилось несколько зависших сообщений — иначе они не дойдут до адресата вообще.

### Шифратор/дешифратор для крота (`cipher_tool.py`)

Отдельная утилита, не часть станции — запускается своим процессом, в том числе на ПК4:
```
python3 cipher_tool.py
```
Спрашивает по порядку: алгоритм (Цезарь/XOR) → ключ → зашифровать или расшифровать → сообщение,
печатает результат и ждёт следующее (можно сменить алгоритм/ключ командой `s`, выйти — `q`).
Считает тем же кодом, что и станции-шифраторы (`common/crypto.py`), поэтому результат гарантированно
совпадает с тем, что реально прошло бы через pk2/pk6. Сценарий использования в ручном режиме: крот
видит перехваченное сообщение (`pending`/`show <id>`), при необходимости прогоняет замену через
`cipher_tool.py`, чтобы она выглядела правдоподобно зашифрованной (или расшифровывает перехваченное,
чтобы понять, что подменить), и вставляет готовый текст в `release <id> <текст>`.

### Спросить мастера (`ask_master.py`)

Отдельный инструмент, не команда станции — запускается СВОИМ процессом в соседнем окне терминала,
рядом с уже работающей станцией:
```
python3 ask_master.py --config configs/pk4.json
```
(или без конфига: `python3 ask_master.py --station pk4 --master-host <IP> --master-port 7000`).
Весь диалог с мастером — и вопрос, и ответ — остаётся В ЭТОМ ЖЕ окне: скрипт держит своё
отдельное постоянное соединение с мастером и регистрируется там как «окно вопросов» для
`station_id`, а не как сама станция, поэтому не может её вытеснить или оборвать. Ответ мастера
(`event <id_станции> <текст>`) приходит именно сюда, а не в консоль станции — пока это окно
открыто, консоль станции такие точечные ответы больше не получает (широковещательные
`event all` по-прежнему идут станциям как обычно, этого окна не касаются). Раньше это была
встроенная команда `ask` в консоли станции — вынесена в отдельный скрипт, чтобы диалог с
мастером не путался с живой лентой входящих сообщений станции.

## Команды мастера

```
list                      — список подключённых станций и их статус
event all <текст>         — отправить событие всем станциям (например тревогу об утечке)
event <id_станции> <текст> — отправить событие конкретной станции
quit
```

## Известные ограничения

- Дашборд мастера перерисовывает экран при каждом новом событии — если в этот момент вы печатаете команду, строка визуально «съедется». Это косметика: просто дописывайте команду и жмите Enter, ввод не теряется.
- Если станция потеряла связь с мастером, она продолжает нормально работать в цепочке (мастер только наблюдает и не участвует в передаче сообщений) — просто перестанет обновляться в дашборде до переподключения.
